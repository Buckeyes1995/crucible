# Crucible — Test Plan 2026-04-21

Continuation of the hands-on evaluation that started in `TEST_PLAN_2026_04_20.md`.
This document covers the 20 smaller enhancements and 5 major enhancements shipped overnight
on 2026-04-20 → 2026-04-21.

**How to test:** walk through each item in order. Mark **pass** / **fail** and include a
one-liner of what you saw. For failures, grab the browser console and the backend

## Part A — Smaller enhancements

### A1. Chat session tags & pinned (DB + API)
**Goal:** chat history can be pinned to the top and labelled with tags.

- `PUT /api/chat/sessions/{id}/pinned` flips the pin flag.
- `PUT /api/chat/sessions/{id}/tags` replaces the tag list.
- `GET /api/chat/sessions` returns `pinned: bool` and `tags: string[]`.

**Steps:**
1. Open `/chat/history`, hover a row, click the pin icon next to the title.
2. Reload — the pinned row should float to the top.
3. Click the same pin again to unpin.

**Expected:** pin stays across reloads. Tags show as small chips below the title when present.

### B1. Inline Markdown renderer in chat

### B2. Benchmark diff report
**Goal:** side-by-side of two runs with per-cell deltas.

**Steps:**
1. Ensure you have ≥2 benchmark runs in history (same models/prompts ideal).
2. Sidebar → Benchmark → **Diff**.
3. Pick two runs; click Compare.
4. Verify table per model with columns A/B/Δ for tok/s and TTFT.
5. Green = improvement, red = regression, grey = within ±2%.

**Expected:** Δ tok/s positive-is-green (higher better). Δ ttft positive-is-red (lower better).

### B3. Mid-conversation model swap
**Goal:** change model without losing chat history.

**Steps:**
1. Load a model (say gpt-oss-20b) and chat a couple of turns.
2. In the chat header, the small font-mono chip is now a dropdown — pick a different model.
3. TopBar shows loading spinner while swapping; chat history stays intact.
4. Send another message — reply comes from the new model.

**Expected:** no page reload, no conversation clear. The new reply is from the chosen model.

### B4. First-run wizard
**Goal:** gentle onboarding for new users.

**Steps:**
1. `localStorage.removeItem("crucible.first-run-dismissed")` in browser devtools → hard reload.
2. Wizard should appear with 5 steps: Models, Chat, Benchmark, Arena, Power-user tips.
3. Navigate with Next, or click a CTA ("Browse Models") to jump.
4. Either Done or Skip dismisses it.
5. Reload — wizard does NOT reappear.

**Expected:** wizard only shows once. Dismiss is permanent unless you clear the flag.

---

### B5. API-key route scoping
**Goal:** scoped keys with permissions (read/chat/benchmark/admin), plus the legacy single-key path unchanged.

**Setup — create a scoped key:**
```bash
curl -X POST http://localhost:7777/api/api-keys \
    -H 'Content-Type: application/json' \
    -d '{"name":"chat-only","permissions":["chat"]}'
# → returns {"key": "cruc_...", ...} — copy the raw key once
```

**Verify from a non-local IP (e.g. phone on LAN, or `curl --resolve` trick):**
```bash
# reach /api/status (scope=read) with chat-only key → 401
curl -H "X-API-Key: cruc_xxx" https://crucible.buckeyes1995.com/api/status
# reach /v1/chat/completions (scope=chat) with chat-only key → passes
# reach /api/settings PUT (scope=admin) → 401
```

**Expected:** chat works, admin-only mutations rejected. Legacy single `api_key` in config.json still grants full access.

**Notes:**
- Localhost bypasses auth (unchanged).
- Tunnel/Cloudflare traffic arrives from non-local, so enforcement kicks in there.
- Expired keys are rejected.

## Known caveats

- **Markdown renderer** uses `dangerouslySetInnerHTML` after HTML-escaping content. If the model emits raw HTML, it renders as literal text (by design). Only whitelisted inline elements (bold/italic/code/links/headings/lists/blockquotes) are transformed.
- **Scoped API keys** still require the legacy middleware pattern; there's no per-key rate limit yet — that's on the v3 roadmap (#149 follow-up).
- **First-run wizard** uses localStorage, so it'll reappear once per browser. That's intentional — a new device/profile is a new user.
- **Disk banner** uses a 10 GB threshold. Adjust in `backend/routers/misc.py::disk_summary`.
- **Audit entries** are best-effort: the record function swallows IO errors so it never blocks the caller.
